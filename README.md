# Мониторинг рабочих ссылок через RU IP

Скрипт раз в N минут проходит по всей цепочке редиректов каждой ссылки, ищет на
конечной странице заданное **ключевое слово** и, если слова нет (или сайт
недоступен), моментально шлёт алерт в Telegram-чат. Проверка обязательно должна
идти с **российского IP**, иначе блокировки РКН не видны.

## Как это работает

- Проверка идёт через **headless-браузер (Playwright/Chromium)**, а не обычным
  HTTP-запросом. Это важно: рабочие ссылки часто ведут на анти-РКН гейтвеи,
  которые выбирают живое зеркало и редиректят на него **через JavaScript** (с
  задержкой в несколько секунд). Обычный `curl`/`fetch` видит только страницу-
  заглушку `Redirecting…` и ключевого слова там нет. Браузер реально исполняет
  JS, проходит редирект до конечного зеркала и рендерит SPA.
- Проверка по статус-коду 200 **не используется** — заглушки провайдеров тоже
  отдают 200. Критерий «жива ли ссылка» = появление ключевого слова в DOM
  конечной страницы. Скрипт ждёт слово до `settleMs` (по умолч. 25 сек) и
  резолвится сразу, как оно появилось; если не появилось — ссылка считается
  забаненной/заглушкой.
- Алерт приходит **только при смене состояния** (рабочая → упала), а не каждый
  час, — чтобы не спамить. При восстановлении приходит отдельное уведомление
  (можно выключить через `alertOnRecovery`).
- Состояние хранится в `state.json` — переживает перезапуск сервиса.

## Конфиг — `config.json`

Всё управление здесь, код трогать не нужно. Файл перечитывается на каждом цикле,
поэтому правки применяются на лету (перезапуск сервиса не требуется).

```json
{
  "telegram": { "botToken": "123:ABC...", "chatId": "-1001234567890" },
  "proxy": "",
  "intervalMinutes": 60,
  "settleMs": 25000,
  "links": [
    { "name": "Joycasino", "url": "https://...", "keyword": "joycasino" },
    { "name": "Dragon",    "url": "https://...", "keyword": "dragon" }
  ]
}
```

- **`intervalMinutes`** — интервал проверки (динамический, по умолчанию 60).
- **`settleMs`** — сколько ждать появления ключевого слова после захода (для
  JS-редиректов и рендера SPA), по умолчанию 25000 мс.
- **`proxy`** — если пусто, идёт прямое соединение (= RU IP сервера). Можно
  указать российский прокси: `http://user:pass@host:port`. Если прокси задан,
  перед каждым циклом идёт **пред-полётная проверка**: скрипт убеждается, что
  прокси пропускает трафик и выясняет выходной IP/страну. Если прокси мёртв —
  цикл пропускается и приходит один алерт «прокси не работает» (а не 4 ложных по
  каждой ссылке). Если страна выхода ≠ RU — предупреждение в лог. Проверить
  вручную: `node index.js --check-proxy`.
- **`links[]`** — добавление/удаление/правка ссылок и ключевых слов. `keyword`
  ищется без учёта регистра.

Добавить ссылку = дописать объект в массив `links`. Удалить = убрать объект.

## Запуск

```bash
npm install                          # зависимости (patchright)
npx patchright install chromium      # скачать браузер (один раз)
cp config.example.json config.json   # создать рабочий конфиг из шаблона
#   → заполнить config.json: botToken, chatId, свои ссылки
node index.js --test-telegram        # проверить, что бот пишет в чат
node index.js --check-proxy          # проверить прокси (выходной IP и страна)
node index.js --once                 # один прогон всех ссылок и выход
node index.js                        # демон: проверка сразу и далее по интервалу
```

> `config.json` содержит секреты (токен бота) и в git не коммитится — он в
> `.gitignore`. В репозитории лежит только `config.example.json`.

## Развёртывание на RU-сервере (Zomro)

Выбран вариант **отдельный дешёвый VPS с RU IP** (стабильный российский IP —
надёжнее ротации прокси).

```bash
# 1. Node.js 18+
curl -fsSL https://deb.nodesource.com/setup_20.x | bash - && apt install -y nodejs

# 2. Код + браузер
mkdir -p /opt/check_link && cd /opt/check_link
#   (скопировать сюда index.js, config.json, package.json)
npm install
npx patchright install --with-deps chromium   # браузер + системные библиотеки

# 3. Создать и заполнить конфиг (токен, chatId, реальные ссылки)
cp config.example.json config.json && nano config.json

# 4. Автозапуск через systemd
cp check_link.service /etc/systemd/system/
systemctl daemon-reload
systemctl enable --now check_link
journalctl -u check_link -f        # смотреть логи
```

## Настройка Telegram (нужно сделать один раз)

1. В Telegram: **@BotFather** → `/newbot` → получить `botToken`.
2. Создать рабочую группу, добавить туда бота.
3. Узнать `chatId`: добавить в группу **@getidsbot** (или @RawDataBot), он
   покажет id (для групп — с минусом, напр. `-1001234567890`).
4. Вписать `botToken` и `chatId` в `config.json`, проверить:
   `node index.js --test-telegram`.

## Ограничения

- Ловятся редиректы любого типа: HTTP 3xx, `meta refresh`, JS
  (`window.location`) и анти-РКН гейтвеи, выбирающие живое зеркало.
- Ключевое слово ищется без учёта регистра во всём DOM конечной страницы
  (включая инлайн-данные SPA). Выбирайте слово, которое реально присутствует на
  рабочем сайте (напр. бренд), но которого не будет на заглушке провайдера.
- Если сайт очень медленный, увеличьте `settleMs`.
